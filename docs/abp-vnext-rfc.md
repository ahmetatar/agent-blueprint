# ABP vNext RFC

## Objective

Evolve ABP from a validated YAML-to-code generator into a declarative, framework-agnostic, PRD-ready workflow system for building robust multi-agent pipelines with:

- high signal-to-noise ratio
- deterministic execution surfaces
- explicit contracts
- replayable and testable runs
- portable orchestration semantics
- artifact-driven outputs

This RFC proposes the next product layer above the current ABP core:

- current core: `schema -> validate -> compile IR -> generate -> run`
- next layer: `spec -> contracts -> policies -> workflow semantics -> harness -> eval -> artifacts -> replay`

## Why

ABP already solves important problems well:

- declarative YAML source of truth
- framework-agnostic IR
- validated graph and tool definitions
- code generation for LangGraph
- local execution and deployment packaging

But the current system still behaves like a graph compiler more than a workflow operating model.

The main gap is not YAML authoring. The main gap is operational determinism and PRD-readiness:

- what exact state is allowed before and after each node
- what output shape is guaranteed
- what tools are allowed under which conditions
- how runs are tested and replayed
- how artifacts are produced and validated
- how regressions are caught before generation or deployment

## Design Principles

### 1. Declarative first

All workflow-critical behavior should be specifiable in YAML without binding ABP to one framework runtime.

### 2. Determinism over convenience

ABP should prefer explicit contracts, declared policies, and reproducible traces over hidden runtime magic.

### 3. Framework-agnostic semantics

ABP should define orchestration semantics once in IR and map them to execution targets. Runtime behavior must not depend on template quirks.

### 4. PRD-ready artifacts

ABP should treat intermediate and final artifacts as first-class outputs, not only chat messages.

### 5. Testability as a product feature

If a workflow cannot be replayed, asserted, and diffed, it is not production-ready.

## Current Gaps

### Declared but weakly enforced

- top-level `input` and `output` schemas exist but are not enforced as runtime contracts
- `human_in_the_loop` and `requires_approval` exist in models but are not enforced in execution
- `checkpoint_every` and `max_graph_steps` exist in settings but are not fully expressed in runtime behavior
- `output_schema` is not a true structured output contract

### Declared but not fully implemented

- `parallel` node type
- `subgraph` node type
- richer fallback and retry semantics

### Missing product surfaces

- agent harness
- eval datasets
- replay/golden traces
- artifact contracts
- workflow policy layer
- compiler linting for determinism and robustness

## ABP vNext Surface Area

The following new top-level sections are proposed.

### `contracts`

Defines runtime expectations and invariants.

Purpose:

- validate state transitions
- constrain node inputs and outputs
- prevent implicit behavior
- make execution auditable

Proposed shape:

```yaml
contracts:
  state:
    required_fields:
      - messages
    immutable_fields:
      - request_id
    invariants:
      - "state.step_count <= settings.max_graph_steps"
      - "state.route in [null, 'billing', 'support', 'sales']"

  nodes:
    router:
      requires:
        - messages
      produces:
        - route
      forbids_mutation:
        - final_answer

    writer:
      requires:
        - research_findings
      produces:
        - final_answer
      output_contract: final_answer_contract

  outputs:
    final_answer_contract:
      type: object
      required: [answer, confidence]
      properties:
        answer:
          type: string
        confidence:
          type: number
```

### `policies`

Defines safety, approval, routing, and execution constraints.

Purpose:

- reduce noise
- constrain dangerous tool usage
- enforce approval paths
- declare escalation rules

Proposed shape:

```yaml
policies:
  approvals:
    mode: selective
    tools:
      - issue_refund
      - send_email
    on_violation: block

  tool_usage:
    max_calls_per_node: 3
    max_calls_per_run: 12
    require_explicit_arguments: true
    on_unknown_tool: fail

  escalation:
    on_low_confidence: handoff_review
    confidence_threshold: 0.75

  budgets:
    max_tokens_per_run: 30000
    max_latency_seconds: 45
    max_cost_usd: 1.5
```

### `artifacts`

Defines named work products generated by the workflow.

Purpose:

- make PRD and delivery outputs first-class
- standardize handoff outputs across agent workflows
- decouple artifact generation from chat-only assumptions

Proposed shape:

```yaml
artifacts:
  prd:
    format: markdown
    contract: prd_contract
    producer: writer
    path: "artifacts/prd.md"

  story_map:
    format: json
    producer: planner
    path: "artifacts/story-map.json"

contracts:
  outputs:
    prd_contract:
      type: object
      required:
        - title
        - problem
        - goals
        - non_goals
        - user_stories
        - acceptance_criteria
```

### `harness`

Defines deterministic scenario-based execution tests.

Purpose:

- validate workflow behavior against expected routes, tool calls, state, and outputs
- support replay and regression testing
- make quality checks framework-independent

Proposed shape:

```yaml
harness:
  defaults:
    llm_mode: mock
    tool_mode: stub
    seed: 42
    freeze_env:
      - OPENAI_API_KEY
    normalize:
      whitespace: true
      timestamps: true
      ids: true

  scenarios:
    - id: refund_happy_path
      input:
        message: "I want a refund for invoice 123"
      expected:
        route: billing
        tools_called:
          - lookup_invoice
          - issue_refund
        output_contract: refund_response
        state_assertions:
          - "state.route == 'billing'"
          - "state.resolved == true"
        artifacts:
          - none

    - id: low_confidence_escalation
      input:
        message: "Please do something unclear"
      expected:
        route: handoff_review
        approvals_triggered: true
```

### `evals`

Defines dataset-driven evaluation suites.

Purpose:

- distinguish harness tests from benchmark-style evals
- measure routing accuracy, artifact quality, and policy compliance

Proposed shape:

```yaml
evals:
  suites:
    - id: router_accuracy
      metric: exact_match
      dataset: datasets/router_cases.yaml

    - id: tool_policy_compliance
      metric: policy_violations
      dataset: datasets/policy_cases.yaml

    - id: prd_quality
      metric: rubric
      dataset: datasets/prd_cases.yaml
```

## Workflow Semantics To Add

ABP needs first-class orchestration semantics beyond linear and conditional edges.

### Parallel fan-out and join

Use cases:

- planner fans out work to multiple specialists
- review by multiple evaluators
- compare strategies before selecting one

Proposed shape:

```yaml
graph:
  nodes:
    gather_inputs:
      type: parallel
      branches: [research, compliance, pricing]
      join: merge_findings
```

IR requirements:

- explicit branch IDs
- explicit join strategy
- defined merge rules for state reducers
- timeout and partial-failure policy

### Subgraph

Use cases:

- reusable review loop
- reusable approval flow
- reusable PRD generation pipeline

Proposed shape:

```yaml
graph:
  nodes:
    prd_pipeline:
      type: subgraph
      ref: prd_generation_v1
```

Subgraph requirements:

- explicit input mapping
- explicit output mapping
- versioned reusable workflows
- isolated internal state namespace or declared merge rules

### Retry and fallback

Use cases:

- low-confidence reroute
- tool failure retry
- model fallback
- policy fallback to human review

Proposed shape:

```yaml
graph:
  nodes:
    researcher:
      agent: researcher
      retry:
        max_attempts: 2
        backoff_seconds: 1
      fallback:
        to: reviewer
        on:
          - tool_error
          - contract_violation
```

### Loop guardrails

Use cases:

- controlled ReAct loops
- bounded critique/refine cycles

Requirements:

- max iterations
- explicit exit conditions
- deterministic state mutations
- loop trace visibility

## Structured Output Model

Current `output_schema` should evolve into a true structured output contract.

Goals:

- support strict JSON/object outputs
- validate outputs at runtime
- fail or reroute on contract violation
- separate message text from workflow outputs

Proposed direction:

- keep `output_schema` as a shorthand
- compile it into a formal contract object
- support node-level output validation
- support top-level final output validation

Example:

```yaml
agents:
  router:
    model: "openai/gpt-4o"
    output:
      contract: route_decision
      strict: true

contracts:
  outputs:
    route_decision:
      type: object
      required: [route, confidence]
      properties:
        route:
          type: string
          enum: [billing, support, sales]
        confidence:
          type: number
```

## Agent Harness

Agent Harness should become a first-class ABP feature, not an external testing convention.

### What it must do

- run scenarios against a blueprint
- support mock, live, and replay modes
- stub tools deterministically
- assert route, tool sequence, state transitions, output, and artifacts
- emit stable traces
- support regression diffs

### Execution modes

- `mock`: LLM and tools both mocked
- `stubbed`: LLM mocked or replayed, tools stubbed
- `live-tools`: live tools with mocked/replayed LLM
- `live`: full live execution
- `replay`: no live execution, compare against stored trace

### Stable assertions

- exact route
- ordered tool calls
- max tool calls
- state assertions
- artifact existence
- output contract validity
- policy compliance

### Trace model

Each run should emit a manifest:

```yaml
run:
  run_id: "2026-04-28T12:00:00Z_refund_happy_path"
  blueprint: "customer-support"
  blueprint_version: "1.2"
  scenario_id: "refund_happy_path"
  mode: "replay"
  seed: 42

trace:
  - node: router
    input_state_hash: "..."
    output_state_hash: "..."
    route: "billing"
  - node: billing
    tools_called:
      - name: lookup_invoice
        args_hash: "..."
```

### CLI surface

Proposed commands:

- `abp test <blueprint>`
- `abp test <blueprint> --scenario refund_happy_path`
- `abp test <blueprint> --mode replay`
- `abp replay <trace-file>`
- `abp trace <run-dir>`

## Compiler and Validation Improvements

`abp validate` should remain schema and reference validation.

Add `abp lint` for workflow correctness and determinism analysis.

### `abp lint`

Checks:

- unreachable nodes
- dead state fields
- condition overlap ambiguity
- missing default branch on conditional routing
- mutable fields changed without reducer semantics
- unsafe loop without guardrails
- approval policy declared but not reachable
- output contract declared but not consumed
- artifact declared but never produced
- non-deterministic prompt hints

### `abp doctor`

Checks runtime portability:

- missing env vars
- unresolved tool impl imports
- unsupported target features
- provider config mismatch
- memory backend mismatch

## Runtime Changes Required

### IR

The IR must carry more than graph shape.

Add:

- node contracts
- policy references
- retry/fallback metadata
- branch/join metadata
- artifact declarations
- harness metadata

### Runtime events

The generated runtime must emit structured events:

- node_started
- node_finished
- tool_called
- tool_failed
- approval_requested
- approval_granted
- contract_failed
- artifact_written
- run_finished

### State discipline

ABP should move from free-form state mutation toward declared mutation rules:

- field ownership
- reducers
- immutability flags
- per-node allowed writes

## Artifact-Driven PRD Workflow

ABP should directly support PRD-ready multi-agent workflows, not only general conversation systems.

Reference workflow:

1. intake agent parses request
2. analyst agent clarifies problem
3. researcher agent gathers context
4. planner agent drafts scope and stories
5. reviewer agent checks completeness
6. writer agent produces PRD artifact
7. policy gate validates required sections
8. final artifact saved and versioned

This is where ABP can become much stronger than a generic graph wrapper.

### Recommended first-class artifact types

- `prd`
- `story_map`
- `acceptance_criteria`
- `research_report`
- `decision_log`
- `handoff_packet`

## Example End-State Blueprint

```yaml
blueprint:
  name: "prd-factory"
  version: "2.0"

settings:
  default_model: "openai/gpt-4o"
  max_graph_steps: 12

state:
  fields:
    messages:
      type: "list[message]"
      reducer: append
    route:
      type: string
      default: null
    research_notes:
      type: string
      default: null
    prd:
      type: object
      default: null

contracts:
  state:
    required_fields: [messages]
  nodes:
    router:
      produces: [route]
    writer:
      requires: [research_notes]
      produces: [prd]
      output_contract: prd_contract
  outputs:
    prd_contract:
      type: object
      required: [title, problem, goals, user_stories, acceptance_criteria]

policies:
  tool_usage:
    max_calls_per_run: 8
  approvals:
    mode: selective
    tools: [publish_prd]

artifacts:
  prd_doc:
    format: markdown
    producer: writer
    contract: prd_contract
    path: "artifacts/prd.md"

harness:
  defaults:
    llm_mode: replay
    tool_mode: stub
    seed: 42
  scenarios:
    - id: prd_generation_happy_path
      input:
        message: "Create a PRD for AI invoice dispute assistant"
      expected:
        output_contract: prd_contract
        artifacts: [prd_doc]
        state_assertions:
          - "state.prd is not null"

agents:
  router:
    model: "${settings.default_model}"
  researcher:
    model: "${settings.default_model}"
    tools: [search_kb]
  writer:
    model: "${settings.default_model}"

graph:
  entry_point: router
  nodes:
    router:
      agent: router
    research:
      agent: researcher
      retry:
        max_attempts: 2
    writer:
      agent: writer
    review_gate:
      type: handoff
  edges:
    - from: router
      to: research
    - from: research
      to: writer
    - from: writer
      to: END
```

## Acceptance Criteria For vNext

ABP vNext should satisfy the following:

- a blueprint can declare runtime contracts and have them enforced
- a workflow can be tested without live providers
- a run can be replayed deterministically from stored traces
- a PRD-producing multi-agent workflow can emit typed artifacts
- policy violations can fail a run before deployment
- unsupported semantics fail loudly instead of silently degrading
- target capability differences are explicit

## Bottom Line

The most valuable next step is not more generator breadth. It is workflow rigor.

ABP should become the declarative control plane for agent systems:

- contracts for correctness
- harness for determinism
- policies for safety and low-noise operation
- artifacts for PRD-ready delivery
- replay and eval for confidence

If this direction is adopted, ABP stops being mainly a YAML code generator and becomes a real specification system for multi-agent production workflows.
